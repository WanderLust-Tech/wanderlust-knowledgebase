# React Reader (`chrome://reader`)

Migration of the Polymer/Web-Components RSS reader at
[`src/custom/browser/resources/reader/`](../../src/custom/browser/resources/)
into a React + Tailwind WebUI under
[`src/custom/components/custom_reader/`](../../src/custom/components/custom_reader/),
mirroring the [`custom_settings`](README.md) pattern.

The same `ReaderUI` controller (no host change, no patch change) now serves
the React bundle — `chrome://reader` is unchanged from the user's perspective.

## What got swapped

| Concern | Before (Polymer) | After (React) |
|---|---|---|
| Page shell | `reader.html` + `reader.js` (~54 KB hand-written JS) | `index.html` + esbuild-bundled `custom_reader.{js,css}` |
| Components | `reader-app`, `reader-feed-list` (Polymer custom elements) | `App.tsx`, `Sidebar.tsx`, `ItemList.tsx` |
| IPC out of JS | `cr.sendWithPromise(...)` calling globals on a `Reader.eventHandler` object | `cr.sendWithPromise(...)` resolved by `ResolveJavascriptCallback` |
| Change notifications | `CallJavascriptFunctionUnsafe("Reader.eventHandler.onFeedListUpdated", ...)` | `FireWebUIListener("readerFeedsChanged")` + React refetches |
| Resources | `IDR_READER_HTML`, `IDR_READER_JS` in [`custom_resources.grd`](../../src/custom/browser/resources/custom_resources.grd) | `custom_reader_resources.grd` (generated by `build_react_webui`) |
| Localized strings | ~30 `IDS_READER_*` keys plumbed into the data source for `$i18n{}` substitution | UI labels live as TSX literals. One exception: `IDS_READER_DISABLED_MESSAGE` is sent through the DOM handler in the `requestReaderState` response so the disabled-state screen stays localized. The remaining `IDS_READER_*` entries in [`generated_resources.grdp`](../../src/custom/app/generated_resources.grdp) are kept for now (used elsewhere, low cost) |

| [`rss_feed.h`](../../src/custom/browser/rss/rss_feed.h) + [`rss_impl.{h,cc}`](../../src/custom/browser/rss/rss_impl.cc) — observer chain | Added `AddObserver` / `RemoveObserver` to `RSSFeed` and a `base::ObserverList<RssServiceObserver>` in `RSSImpl`. `NotifyRSSChanged` now actually iterates observers (the old code self-called a stub) and hops to the UI thread when invoked from `RequestRSSCallback`. `ReaderDOMHandler` registers itself in `RegisterMessages` so external mutations (e.g., infobar subscribe) push into the reader without a reload. |
| [`rss_impl.cc`](../../src/custom/browser/rss/rss_impl.cc) — lock discipline | Several methods used to do `cache = std::move(cache_)` under `lock_` and then call public `SetRSSChannel` / `SetRSSGroup` (which re-acquire `lock_`) → recursive-lock deadlock plus a permanent `cache_ == nullptr`. | [`reader_dom_handler.cc`](../../src/custom/browser/ui/webui/reader/reader_dom_handler.cc) — `profile_` init | Stores `profile_` in `RegisterMessages` (which runs during WebUI setup, before any JS) instead of in `OnPageLoaded` (which fires after React's effects). Child-effect-before-parent-effect ordering meant `useFeedList`'s first `requestFeedList` used to land while `profile_` was still null, returning an empty list every time you reopened `chrome://reader`. |
| [`rss_fetcher.cc`](../../src/custom/browser/rss/rss_fetcher.cc) — parser robustness | The XML state machine now records the root element's reported depth on first sight and uses `root_depth + N` for all downstream depth checks (libxml's reader inconsistently reports root depth 0 vs 1). RSS 2.0 / RSS 1.0 / Atom roots are recognized by name regardless of depth. Added `content:encoded` (RSS 2.0 extension) and `summary` / `published` (Atom) as recognized description / date fields. `ReadElementContent` was swapped for a local `ReadElementTextNoAdvance` helper that leaves the reader **on** the closing tag — the original advanced past it, which combined with the outer-loop's `Read()` to eat every other sibling element. ## IPC contract

The C++ side stays canonical for the IPC name + payload shape. React side
just consumes it via the `cr` polyfill.

The sidebar WebUI (see [sidebar.md](sidebar.md)) consumes the same
IPC names from a separate `SidebarDOMHandler` so its compact RSS view
shares the `RSSImpl` backend with chrome://reader. `readerFeedsChanged`
and `readerUnreadChanged` are fired by both handlers, so subscribing /
opening a feed in one host updates the other live.

### Reads — `cr.sendWithPromise(name, ...args)` → `ResolveJavascriptCallback`

| Message | Args (after callback id) | Resolves with |
|---|---|---|
| `requestReaderState` | — | `{ enabled: boolean, disabledMessage: string }` (message is `IDS_READER_DISABLED_MESSAGE`) |
| `requestFeedList` | — | `{ feeds: Feed[], groups: Group[] }` |
| `requestFeedContent` | `channelUrl: string` | `FeedItem[]` (also marks the channel's items as read) |
| `requestFeedContentGroup` | `groupKey: string` | `FeedItem[]` (also marks group's items as read when `groupKey != "0"`) |
| `requestFeedContentBySearch` | `keyword: string` | `FeedItem[]` filtered by title substring |

### Writes — `chrome.send(name, args)` fire-and-forget

| Message | Args |
|---|---|
| `onPageLoaded` | `[]` — handler calls `AllowJavascript()` and flips its `page_loaded_` flag |
| `addFeed` | `[feedUrl, groupKey]` |
| `removeFeed` | `[feedUrl]` |
| `updateFeed` | `[feedUrl, title, groupKey, sort]` |
| `addGroup` | `[title]` |
| `removeGroup` | `[groupKey]` |
| `updateGroup` | `[groupKey, title, sort]` |
| `showOPMLImportDialog` | `[]` |
| `showOPMLExportDialog` | `[]` |
| `initialize` | `[]` — resets to default RSS channels |
| `refreshAllFeeds` | `[]` — queues every channel for an immediate fetch via `RSSFeed::RefreshAllRSS()`, bypassing per-channel `update_interval` |
| `createBookmark` | `[title, url]` — currently a no-op, see [Deferred](#deferred) |

### Listeners — `cr.addWebUIListener(name, fn)` ← `FireWebUIListener`

| Event | Payload | Sent when |
|---|---|---|
| `readerFeedsChanged` | — | `RSSServiceObserver::OnRssChanged()` fires on the service |
| `readerStateChanged` | `{ enabled, disabledMessage }` | `prefs::kRSSEnabled` flips. Lets `App.tsx` swap to/from the disabled-message screen without a reload. |
| `readerUnreadChanged` | `{ channelUrl, unreadCount }` | A view-driven mark-as-read finished (per-channel after `requestFeedContent`, one per affected channel after `requestFeedContentGroup` with a non-zero group). `useFeedList` patches the matching feed's `unreadCount` in place so the sidebar badge clears immediately instead of waiting for the next full refetch. |

## Data shapes

```ts
interface Feed {
  url: string;          // channel URL — also the primary key
  title: string;
  link: string;         // human-facing site URL
  description: string;
  sort: number;
  groupKey: string;     // stringified int64 (matches RSSChannelInfo::group_id)
  unreadCount: number;  // computed server-side per request, not stored
}

interface Group {
  key: string;          // stringified int64
  title: string;
  sort: number;
}

interface FeedItem {
  channelUrl: string;
  title: string;
  link: string;
  description: string;  // HTML; sanitized server-side for U+2028/U+2029.
                        // ItemList.tsx strips tags via DOMParser before
                        // rendering — chrome://reader's CSP disallows
                        // innerHTML, and dumping feed-author HTML on a
                        // privileged scheme would be an XSS foot-gun.
  posixTime: number;    // ms since epoch
  relativeTime: string; // pre-formatted, e.g. "3 hours ago"
  read: boolean;
  bookmarked: boolean;
}

interface ReaderState {
  enabled: boolean;        // mirrors prefs::kRSSEnabled
  disabledMessage: string; // localized — IDS_READER_DISABLED_MESSAGE
}
```

## What the UI exposes today

Not every IPC message above has a corresponding affordance in the React
shell. The flip side of [Deferred](#deferred) — what users can actually
click — is:

**Sidebar** ([`Sidebar.tsx`](../../src/custom/components/custom_reader/components/Sidebar.tsx))
- **↻ Refresh** button in the header → `refreshAllFeeds`. Queues every channel for an immediate fetch (the backend chains through them sequentially, one every `kRSSPeekSequenceSeconds`); the button shows a 3-second spin to acknowledge the click since the backend has no completion signal.
- **Search articles** input (debounced 250ms) → `requestFeedContentBySearch`. While a keyword is set the right pane shows the matched items; clearing the box returns to the previously-selected feed or group view.
- **Add feed** input + a folder `<select>` → `addFeed(url, groupKey)`. Defaults to "No folder" (the Ungrouped bucket).
- **+ New folder** → inline form → `addGroup(title)`.
- **All feeds** row at the top → `requestFeedContentGroup('0')`, shows an aggregate unread badge summed from `Feed.unreadCount`.
- **Per-folder row**: click the header (or chevron) to collapse/expand; collapse state persists across reloads via `localStorage` (`customReader.collapsedFolders`). Hover reveals **✎ rename** (inline edit → `updateGroup`) and **✕ delete** (with a confirm prompt → `removeGroup`). Double-click the header also enters rename mode.
- **Per-feed row**: 16px favicon from `chrome://favicon2?pageUrl=…` (served by `FaviconSource` registered in `reader_dom_handler.cc`). Click selects the feed; hover reveals **✎ edit** (inline panel with title + folder picker → `updateFeed`, sort preserved) and **✕ delete** (confirm prompt → `removeFeed`).
- Per-feed unread badge from `Feed.unreadCount`.
- Footer: **Import OPML** / **Export OPML** buttons, plus **Reset…** (confirm prompt → `initialize`) on the right.

**Item list** ([`ItemList.tsx`](../../src/custom/components/custom_reader/components/ItemList.tsx))
- Three view layouts toggleable from the header's **☰ / ▤ / ▦** button group (persisted to `localStorage` as `customReader.readType`):
  - **Title** — compact list: title + relative time + plain-text summary clamped to 3 lines.
  - **Magazine** — title + relative time + plain-text summary clamped to 3 lines + a 96×96 thumbnail extracted from the first `<img>` in the description (http/https/protocol-relative).
  - **Full** — bigger title (as its own anchor), inline image up to `max-h-80`, full plain-text description (no clamp), and a footer **Open original article** button. Two anchors at separate DOM levels keep nested-`<a>` invalidity out of the markup.
- All description text goes through `stripHtml` (DOMParser-based) before rendering. The WebUI's CSP enables `require-trusted-types-for 'script'`, which bans `innerHTML` / `dangerouslySetInnerHTML` outright — so we deliberately strip feed-author HTML rather than wire a Trusted Types policy that would let unsanitized third-party markup run on the privileged `chrome://reader` scheme. Re-introducing rich-HTML descriptions later means adding a sanitizing TT policy first.
- In group and search modes (where the header isn't a single feed), each item shows a channel-attribution line — favicon + feed title — above the headline, resolved via a `feedsByUrl` map passed down from `App.tsx`.
- In search mode, the keyword is wrapped in `<mark>` (yellow / dark-yellow) within each item's title. Matching is case-insensitive and regex metacharacters are escaped so a keyword like `C++` is safe.
- Click → opens the article's source URL in a new tab.
- Items returned with `read: true` dim to 60% opacity.
- Header has an **All / Unread / Read** filter button group — purely client-side, filters whatever items the current view supplied (works in feed, group, and search modes). Selection persists across reloads via `localStorage` (`customReader.readFilter`).
- When the sidebar is in search mode, the header swaps to "Search results for …" and shows the match count. Group mode uses the group title (e.g., "All feeds") as the header.

**App shell** ([`App.tsx`](../../src/custom/components/custom_reader/App.tsx))
- Selection state lives as a `ViewMode` union — `{type: 'feed', url}`, `{type: 'group', groupKey}`, or `{type: 'search', keyword}` — driven by the sidebar. The last non-search view is remembered so clearing the search box restores it.
- Disabled-state screen: when `rss.enabled` is off, the entire sidebar + item-list UI is replaced with the localized `IDS_READER_DISABLED_MESSAGE`. Toggling the pref in [`chrome://settings`](../../src/custom/components/custom_settings/components/RssPage.tsx) flips this live via `readerStateChanged`.

Backend handlers that exist but have **no UI yet** (and so are
unreachable from the React app today): per-folder selection (the
`requestFeedContentGroup` path only lights up via "All feeds" — clicking a
folder header toggles collapse, not selection) and `createBookmark`. See
[Deferred](#deferred) for the rationale.

## Migration verification

After a fresh `gn gen` + build, `chrome://reader/` should:

1. Render the React shell (left sidebar, right pane) instead of the
   Polymer layout — the visual change confirms which bundle is serving.
2. The sidebar shows whatever channels `RSSFeed::GetRSS` returns (groups
   first, ungrouped second). If empty, only the "Add feed URL…" input
   and OPML buttons are visible.
3. DevTools → Network: a single `custom_reader.js` and `custom_reader.css`
   load, no `reader.js`/`reader.html`.
4. DevTools → Console: `chrome.send('requestFeedList', [...])` is observable
   when the page first paints and again after `addFeed`.
5. Subscribing via the [RSS infobar](../rss-infobar-subscribe.md) — the
   service fires `OnRssChanged` → handler calls
   `FireWebUIListener("readerFeedsChanged")` → React's `useFeedList`
   refetches → the new channel appears in the sidebar.
6. Subscriptions and fetched items persist across a browser restart.
   `RSSBackend::SetRSSChannel` writes channels through to the SQLite
   `channels` table; `RSSImpl::RequestRSSCallback` mirrors fetched
   items into the `items` table via `backend_->SetRSSItem`. On launch,
   `RSSImpl::OnGotRSSData` rehydrates the in-memory cache from
   `RSSDatabase::GetRSSData`. (Each of these was broken at one point — the
   inverted `if (!db_->Init(path))` check silently null'd the DB on
   success, items were never `backend_->SetRSSItem`'d, and several
   `Set*` methods moved `cache_` out without restoring it. All resolved.)

## Deferred

These were either stubbed in the original Polymer reader or skipped to keep
the migration scoped:

- ~~**Per-feed unread badge accuracy**~~ — resolved. `ReaderDOMHandler` now
  fires `readerUnreadChanged` immediately after `SetRSSItemReadWithChannelURL`
  (one event) and `SetRSSItemReadWithGroupID` (one event per channel in the
  group). `useFeedList` patches the in-memory list in place, so badges clear
  the instant a feed is opened. The next `readerFeedsChanged` still does a
  full refetch on subscription churn.
- **`CreateBookmark`** — stubbed in C++ (was already commented out
  pre-migration; depends on a `BookmarkModel` API that drifted). The React
  side doesn't call it yet; wire `readerActions.createBookmark` once C++
  is implemented.
- **Per-folder selection in the sidebar** — clicking a folder header
  currently only toggles collapse. The backend already supports
  folder-scoped item lists (`requestFeedContentGroup(groupKey)` with a
  non-zero key) and would mark the folder's items as read on view
  (mirroring per-feed behavior). Wire a separate affordance — e.g., a
  click target distinct from the chevron — when this view becomes useful.
- **Article preview pane vs. open-in-tab** — `ItemList.tsx` opens the
  source URL in a new tab on click. Original Polymer reader had an inline
  article preview; restore by adding a third pane if desired.
- **OPML import/export error toasts** — buttons fire the dialog; success/failure
  feedback isn't surfaced to React yet. The handler logs to `chrome_debug.log`.
- **Localized strings** — most TSX still uses English literals; the only
  localized string today is `IDS_READER_DISABLED_MESSAGE`, plumbed through
  the `requestReaderState` IPC response. If we want broader localization,
  plumb the `IDS_READER_*` table through `source->AddLocalizedStrings`
  and read from `loadTimeData` in the React app (rather than expanding
  the IPC payload).
